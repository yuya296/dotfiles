# Git Auto Commit Workflow Reference

このリファレンスは git-auto-commit スキルで扱う 4 ステップ（変更確認→差分要約→コミットメッセージ作成→git commit）を細かく示します。メインの SKILL.md で迷ったときに参照してください。

## 1. 変更確認
- `git status --short --branch` を基本とし、未追跡や競合もそのまま共有する。
- 追加・削除・修正ファイルが混ざる場合はカテゴリ別に整理して伝える。
- 大量の変更がある場合はディレクトリ単位にグループ化し、追加調査が必要な箇所をマーキング。

### `git_context_snapshot.py` の活用
```
scripts/git_context_snapshot.py [--out-dir path] [--diff-args ...]
```
- 標準では `.codex/git-auto-commit/` に `git-status-タイムスタンプ.txt` と `git-diff-タイムスタンプ.patch` を生成。
- 生成されたファイルパスをユーザーと共有するとレビューが容易になる。

## 2. 差分要約（日本語）
- ファイル単位で「何を」「なぜ」変えたかを 1 行でまとめる。
- 影響範囲が広い場合は「機能」「テスト」「その他」のように分類。
- 要約例:
  - `api/order_service.py`: フック追加で決済リトライを抑制。
  - `tests/test_order.py`: 失敗ケースを分岐別に分解し網羅度向上。
- 仕様が不明な場合は仮説として明記（例: 「推測: バリデーションの境界値調整」）。

## 3. コミットメッセージ（日本語）
1. 1 行目に 50 文字以内の要約（文末句読点なし）。
2. 2 行目は空行。
3. 3 行目以降に詳細。`変更理由 / 動作確認` の順に bullet を推奨。

テンプレート:
```
<要約>

- 変更理由: ...
- 動作確認: ...
```

ヒント:
- 変更が多い場合は「〜を整理」「〜を刷新」など広めの語を使い、詳細は bullet で補う。
- バグ修正は再現手順と fix 方法を簡潔に入れる。
- 機能追加はユーザーストーリー視点（例: 「在庫同期ジョブが遅延した際も救済できるようにする」）。

## 4. `git commit` 実行
- 事前に必要なファイルが stage 済みか `git status` で確認。
- 追加ステップがある場合（例: ビルド成果物除外、lint 実行）はコミット前の TODO として書き残す。
- コミット後は `git status --short` を再実行し、クリーンであることを報告。

## トラブルシューティング
- **未追跡ファイルが大量にある**: まずパターンを確認し、不要ファイルは `.gitignore` 提案としてメモ。
- **diff が巨大で読みづらい**: `--diff-args '--stat'` で統計を併記し、要約時は重要ファイルから順に扱う。
- **コミット拒否 (hooks)**: フックのエラー内容をそのまま引用し、再実行のための TODO を書く。
